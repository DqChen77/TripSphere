\chapter{引言}

\section{研究背景}

随着大语言模型（LLM）技术的迅猛发展，AI 技术正越来越多地深度融入业务流程与系统架构。智能化软件在传统软件的基础上，呈现出包含智能体（Agent）、工具链、业务状态、长期记忆以及外部云服务等多元要素深度耦合的系统特征，在当前时代背景下有成为行业主流的趋势。

随着包括但不限于 LangGraph 的智能体编排框架出现，业界已能将智能体执行过程以图结构显式建模。在此支撑下，智能体由提示词驱动，具备理解业务目标、自动规划行动并动态调用多种工具的能力，能够协调传统业务 API、数据状态和实时用户需求，成为复杂流程中的核心执行中心。智能体层也相应出现了传统软件工程中的可靠性、可观测性与故障定位问题，智能体软件的工程化运维与评估也面临了新的挑战。

随着智能化软件在实际场景中的广泛应用，用户在真实使用过程中常常会遭遇多种潜在问题与缺陷。更为重要的是，由于智能化软件具有一定的容错能力，某些缺陷可能被系统部分掩盖——表现为悄然的能力退化（degradation），未必以显式错误暴露出来，更难被及时发现。这使得部分系统缺陷难以暴露，却在真实生产环境中直接影响用户体验和系统稳定性。因此，洞察并主动模拟这些典型的、易被掩盖的缺陷，是提升智能化软件鲁棒性和可靠性的关键前提。

混沌工程为分布式系统可靠性验证提供了重要方法。Netflix Chaos Monkey 及后续 Chaos Engineering 实践强调在受控条件下主动注入故障去模拟系统缺陷，观察系统是否偏离稳态，并据此改进容错能力\cite{basiri2016chaos,netflixChaosMonkey}。然而，传统工具主要面向资源层和基础设施层，例如终止实例、制造网络延迟、施加 CPU 或磁盘压力。它们可以验证微服务在 Pod 失效或网络抖动下是否稳定，却很难精确制造“仅本次请求的工具返回空列表”、“仅某个 LangGraph turn 的状态被清空”这类智能体层故障。

与此同时，智能体运维和大语言模型运维平台正在快速发展。LangSmith、AgentOps、Langfuse 等工具已经能够记录模型调用、工具调用、Token 用量、Trace 路径和 Bad Case；OpenTelemetry 社区也在推进 GenAI 与 Agent Span 的语义约定。\cite{otelAgentObservability,otelGenAIAgentSpans,otelGenAIMetrics}。这些进展说明 AI智能体的可观测性正在标准化。然而，现有 AgentOps 与可观测平台更多聚焦于运行过程记录与事后分析，缺少对智能体应用层故障的主动注入与可复现实验能力。

\section{研究目的与意义}

在智能化软件中，能发现故障并不等于能稳定复现故障。在工程排障、鲁棒性评估和论文实验中，仍需要一个可以声明故障、重复执行、关联 Trace 并回收证据的故障实验平台。本毕业设计研究目的正是在不大规模改写业务代码的前提下，为智能体构建可复现的应用层故障注入机制。本毕业设计设计并实现了一种面向智能体应用层的故障注入机制，该机制不替代传统混沌工程，也不替代 AgentOps 平台，而是连接二者：它在 Agent、工具、状态和 RPC 边界上制造可控异常，并把注入参数、命中证据和系统响应写入可观测数据，使实验能够被复查和复跑。

从工程与实践意义看，本毕业设计给开发者提供一个接近真实业务场景的故障实验环境，并且协助开发者主动复现坏场景，降低 Bad Case 难以重现的问题。通过设计可配置、低侵入的应用层故障注入机制，将应用层故障注入把实验对象从机器、容器和网络资源推进到智能体决策、工具返回、状态流转和跨服务委托等位置。通过收集 Trace 与运行证据识别智能体层发生的静默退化，进而形成可复现的可靠性验证流程。本毕业设计把实验编号、故障 DSL、负载条件、注入命中证据、端到端延迟、Fallback 行为、业务质量变化和恢复情况纳入同一条证据链，使单次请求、单条 Trace 和单个工具调用都可以成为可复查、可复跑的实验对象。

从学术的角度看，本毕业设计把混沌工程从资源层扩展到 LangGraph 应用层，补充了智能体可观测工具在主动实验能力上的不足，也为智能化软件 Benchmark 和鲁棒性评估提供了基础设施。通过把故障定义、注入命中、Trace 证据和实验结果统一到同一套验证流程中，本毕业设计为后续比较不同智能体架构、Fallback 策略和可观测方案提供了可复现依据。

\section{研究内容与贡献}

围绕上述目标，本毕业设计将围绕智能化软件故障注入靶场展开，设计并实现一个名叫 TripSphere 的旅行网站作为可靠性验证系统，并在系统上做了故障注入实验与结果分析。

首先，本毕业设计实现了一个面向旅行规划场景的智能化软件靶场。该靶场以真实旅行网站为基础，向用户提供行程规划、对话式问答和订单处理等功能。在系统内部，智能体会根据用户需求调用不同工具、访问业务服务、维护会话状态，并在必要时把任务委托给其他服务完成。相比只构造简单示例程序的实验环境，该靶场保留了真实业务流程中的多步骤调用、跨组件协作和可观测边界，为后续故障实验提供接近实际系统的运行环境。

其次，本毕业设计设计了一种面向智能化软件的低侵入故障注入机制。该机制允许实验者在不大规模修改业务代码的前提下，针对工具调用、模型调用、状态传递、服务访问和数据读写等位置声明故障，并控制故障只在指定请求或指定实验中生效。系统在故障发生时记录注入参数、命中位置、系统响应、降级行为和恢复情况，使一次故障实验能够被追踪、复查和重复执行。

再次，本毕业设计完成了面向论文实证的实验结果与可靠性分析。在前两个部分的基础上，本毕业设计设计多组对照实验，在正常负载和注入故障的条件下观察系统表现。实验从请求是否成功、响应时间是否变化、业务结果质量是否下降、系统是否触发降级逻辑等角度分析智能化软件的可靠性。通过这些实验，本文重点讨论了表面请求成功但实际能力下降、故障被降级逻辑掩盖、多个组件之间相互放大故障影响等现象。

本毕业设计的贡献不是提出新的智能体算法，而是尝试性地将混沌工程思想、智能体应用可观测能力和真实业务系统结合起来。一方面，本文通过旅行网站靶场把智能体、工具、状态和业务服务组织为可重复运行的实验对象，使智能化软件可靠性验证不再停留在抽象讨论。另一方面，本文通过低侵入的故障声明、注入命中和观测记录机制，使单次请求、单次工具调用和一次系统降级都能够成为可复查、可复跑的实验样本。最终，本文以实验数据和案例研究的方式说明智能化软件中静默退化、降级掩盖和组合故障放大的具体表现，为后续比较不同智能体架构、容错策略和可观测方案提供基础。

\section{论文结构}

本论文结构安排如下：第二章综述混沌工程、故障注入、AgentOps、OpenTelemetry GenAI 可观测性以及 LangGraph 等相关技术背景。第三章结合 TripSphere 系统的主要业务链路，分析系统架构，提出应用层故障模型，详细阐述故障声明 DSL、注入框架、可观测证据链及实验靶场的实现方式。第四章展示不同链路和实验象限下的实验数据，分析故障注入对 HTTP 指标、业务质量、Agent 级可观测性的影响，同时选取部分典型场景进行案例分析。第五章对全文进行总结，讨论当前工作的局限性与未来的研究方向。
